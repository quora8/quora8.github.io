程序员是如何创建计算机语言的？

编程
|阅读量:

回答一

这是一个漫长的过程。机器语言对人来说真的很难写。所以在机器语言中，早期的程序员编写了一个简单的翻译程序，以便他们可以使用简短的助记符而不是数字来表示指令。这会将助记符转换为实际的机器语言。我们称这种简单的语言为“汇编”。

然后使用汇编语言，他们编写了更复杂的程序，这些程序可以读取更复杂的语言并将其转换为机器语言。这就是 FORTRAN 和 C 的诞生方式。

这些读取一种语言并将其转换为另一种语言的程序称为“编译器”。在接下来的三十年左右发生了大量的创新，以使编译器能够更好地生成真正高效的机器语言。

对于大多数编译器来说，最大的问题是它们会产生机器代码，而机器代码只能在为其编写的特定处理器芯片上工作。大多数 PC 中使用的 Intel 80x86 芯片的机器代码与最新 iPhone 中使用的 Apple A16 芯片或许多 Android 手机中使用的 ARM 芯片的机器代码不同。那么如何编写 C 并将其编译到许多不同的处理器芯片上呢？

答案是中级语言。创意设计师创造了一些非常低级的语言，在某些方面类似于汇编程序，但不依赖于任何特定的处理器芯片。编译器执行三个步骤（这非常简化）。编译器将：

将 C、C++、FORTRAN、Ada 或 Go 等实际编程语言翻译成 GENERIC 等中间语言。
然后编译器将优化中间代码，消除冗余并减少内存占用。
然后编译器将调用特定于操作系统和处理器芯片的最后阶段，以创建特定于处理器的机器语言。
请注意，许多流行的语言不会做最后一步。他们实际上并没有将语言编译成机器语言，而是假装自己是处理器并解释中间语言本身。Java、JavaScript、C#、PHP、Python 和 Ruby 都在中间代码级别进行解释。

这些中间语言有不同的名称，但我们很少讨论它们，因为我们没有看到它们。口译员本身也有名字。我们通常将它们称为虚拟机。Java 虚拟机或 JVM 就是一个例子。Microsoft 的公共语言运行时 (CLR) 是另一个。

您可以使用开源 GNU Compiler Collection 编写自己的语言，将您的语言转换为中间语言（并让编译器完成剩下的工作）。GNU CC 将编译成大约 30 种不同的处理器芯片。这一切都是免费的。

或者您可以编写一个编译器，从解释语言列表中写出其中一种中间语言，并让解释器完成实际工作。

不管怎样，做起来都不容易。

Microsoft 有一个完整的 SDK 供开发人员为 CLR 创建语言。以这种方式编写的语言有很长的列表，包括 C#、Iron Python、VB.NET、Eiffel、F#、PeachPie、Swift (Dibol) 和 PowerShell。

当然，您可以对 Java VM 执行相同的操作。JVM 的编译器可用于 Kotlin、Groovy、Scala、Clojure (Lisp)、COBOL、Haskell、Prolog、R、Scheme 等等。

有趣的是，Fantom 语言可以由 JVM 和 CLI 编译。这很有趣。

现在有数百种计算机语言，有很多方法可以编译和运行它们。你不必做所有的工作。

我认为每个学生都应该编写自己的语言，在 GNU CC、CLI 或 JVM 中运行，这将是一门非常好的计算机科学学位课程。

回答二

一开始没有程序员。

一切都是绕线。每个电路都是独一无二的。

今天，这被称为组合逻辑。

然后有人创造了一个流行的电路。

我的意思是拥有或管理资金的企业。

他们需要工程师来验证钱没有丢失。

计算机增加了记忆来处理更复杂的情况。

然后 John von Neumann 建议，与其通过绕线进行编程，不如将操作代码存储在内存中，并且在每个时钟周期，可编程 CPU 将执行由唯一操作代码指定的一件事。可用的指令集定义了机器架构。程序员现在可以用汇编语言编写代码，而不是工程师进行绕线。

一些计算机公司通过复制流行的 IBM 架构但使用不同的时钟速度来竞争。有些更便宜、更慢，但有竞争和创新。新架构被推向市场，最终周期时间接近 100 万赫兹或 1 微秒。

每个计算机供应商都有自己的汇编语言，可以利用该架构的特性，但 fortran 和 Cobol 指向具有架构独立特性的语言。这些语言的编译器并非在所有硬件平台上都普遍可用。这些计算机不仅价格昂贵，而且它们的编译器是单独出售的。

一个知道如何在某个品牌的计算机上编程的人不能将这些技能转移到另一个品牌。最终，银行和其他大型机客户了解到，业务应用程序和程序员输出之间的鸿沟超过了使用汇编语言可以弥合的鸿沟。

AT&T 还需要计算机技术来运行其电话网络，但没有使用 IBM 品牌的产品。它的一位研究人员在 BCPL 之后建模了一种编程语言，它类似于 IBM 的 COBOL 的 PL/I 后代。他称它为C。

所有这些历史都可以这么说：程序员没有定义或创建我们今天使用的编程语言。教授和研究人员（和 CACM）已经确定了这些高级语言所需的特性，甚至包括 c++ 编译器中可用的面向对象技术。

另一方面，gnu 编译器集合为多处理器架构实现了出色的代码生成，并支持众多 c 和 c++ 标准。它的程序员已经实现并改进了该编译器 30 年。真的，没有好的编译器的计算机语言有什么用？我在 2004 年辞掉了一份工作，因为 Microsoft c 编译器对模板的支持很差。

回答三

“计算机语言”是由简单指令的电子工程师创建的，用于将正确的电子组件耦合在一起以实现计算。这就像安排原子建造房子一样。

我们可能会认为，正因为如此，程序员确实发明了砖块——建造房屋的更高层次的抽象。这本身就可能促使程序员创建编程语言。然而，用语言交流需要做的事情的想法早于电子计算机。许多想法实际上是由 Turing 和 Church 在 1930 年代提出的，尽管如果您研究图灵机，您会发现这些是计算的原子。

因此，我们对“计算机”语言不感兴趣，而是对机器和计算机独立的“编程”语言感兴趣。也就是说，问题可以用它们自己的抽象术语来表达。确实，结构化编程引入了计算砖块，取代了 goto 的旧原子。数据结构引入了连接的数据项，在内存地址级别替换了旧的指针原子。

因此，我们已经从用于计算的物理资源转向了逻辑抽象。这就是编程语言应该做的。唉，太多人认为我们在编程计算机，而实际上应该编程问题 - 编程是关于编程，而不是计算机。

第一种语言是由更多的临时设计创建的——Autocode，然后是 FORTRAN 和 COBOL。然后寻求一种更正式的方法，导致 BNF 来描述明确的语法。这导致了 ALGOL。1960 年代，人们对​​具有指称语义和公理语义的语义的正式定义产生了更多的兴趣。

语言本身的编译器是用计算机语言编写的，一旦创建了简单的编译器，就可以使用该简单的编译器来实现该语言。这是自举。一段时间后，我们可以完全忘记计算机语言，甚至是符号汇编。

不幸的是，我们称这个学科为计算机科学。对 CS 的深刻见解是，它不是关于计算机，而是关于编程的理论和基础。

“CS 与计算机无关，就像天文学与望远镜无关”

回答四

首先是机器代码，使用二进制开关将指令一一切换到机器中。
接下来是穿孔卡片和纸带，它们做同样的事情，但更方便。
接下来是低级汇编程序，其中机器代码（操作代码）被首字母缩写词取代。
接下来是可以通过一个命令生成一系列汇编指令的宏。
接下来是库，例如 I/O 控制系统，其中子程序将提供复杂的服务。
宏和库导致了可以提供更合理的命令流的语言的想法。
最初的几种语言是一个多元化的群体：Fortran、COBOL，以及最重要的 ALGOL。
有人尝试制作融合了每种语言的最佳特性的超级语言。它们是 PL/1 和 Algol 68。它们都没有长寿。除了这些语言之外，还为字符串处理、列表处理以及数学和统计的特殊领域创建了其他专用语言。
UNIX 是作为可移植操作系统开发的，该操作系统可以轻松移植到具有不同架构和底层机器代码的不同机器上。
与 UNIX 一起出现的是 C 语言，由程序员创建并为他们自己的方便而设计。它具有最大的实用性和耐用性。

这只是我个人对过去 60 年所见所闻的看法。其他人可能会提供更正或不同的意见。

真正的计算机语言始于能够绕过开关/重新布线并将指令直接输入机器。随着时间的推移，我们在简化编程方面变得更好，记住每次添加时都要检查溢出，这成为通用宏的语言默认设置。现在我们可以只说 print x,y,x 而不是将最后一条指令插入机器来告诉打印机如何定位和打印。

我更喜欢说“嘿，你的衬衫着火了”，而不是指指点点和尖叫哦哦哦。这可能意味着“你的衬衫着火了”、“杀了那只猛犸象”或“当你弄清楚我的意思时，Trex 已经把你当作零食吃了”。

计算机读取机器代码。机器码只是 CPU 接收的一堆数字，然后用来产生某种输出。在编程语言之前，机器代码是手工编写的，非常令人头疼……然后编写了汇编程序。用手。在机器代码中。那时事情开始变得不那么令人头疼了，因为现在你只需要编写简单的（相比之下）人类可读的字母和数字来进行操作和管理内存。机器代码逐个字符地读取汇编代码，以产生所需的输出。如今，如果您完全失去理智，您可以使用机器代码进行编程，但您最好使用 Python、JS 或 C 之类的东西。

回答六

首先要记住的是，编程语言的设计与其实现是分开的。如果您主要对设计语言感兴趣，那么您不必担心实现优化编译器、高效解析的技术，或者老实说，您对编译器课程的大部分期望。背景考虑到这一点，让我们看看如何设计语言。第一步是熟悉不同的编程范式（命令式、函数式、面向对象、逻辑……）并学习一堆不同的语言。对不同的编程语言有广泛的了解
如果你想自己设计，这非常重要：它们会给你灵感，让你知道做什么，也许更重要的是，不做什么（想想松散的打字 à la JavaScript 和无限制的 goto）。我建议至少能够阅读以下大多数语言：
SML可以看到一种优雅的语言，具有经过深思熟虑的语义最简单的方案、出色的元编程和延续JavaScript可以替代 OOP，看看如何不为低层次的抽象加上大量令人困惑的极端案例Haskell了解真正的静态类型和函数式编程的实际应用Prolog体验逻辑编程的好坏更容易利用现有的想法和功能，但实际上也会更容易提出合法的新事物。设计一门语言我们可以将设计一门语言的实际问题分解为三个部分：语法、语义和类型
. 它的真正美妙之处在于，我们实际上可以在设计时在很大程度上分别考虑这些！这也使得随着时间的推移独立更改它们变得更容易。语法
语法可能是设计语言中最简单的部分。然而，这对于大众采用非常重要. 你不应该通过封面来判断一本书，但人们仍然会这样做——对于编程语言和语法也是如此！此外，语法对于鼓励或阻止某些模式也很有用。只需比较常见的编程风格，比如 JavaScript 和 Scheme。在宏观层面上，它们并没有太大的不同。但是因为 JavaScript 在语句和表达式之间有语法上的区别，并且因为使用匿名函数有很多语法开销，所以环境 JavaScript 编程风格是非常不同的（而且，我想说，更糟) 来自常见的 Scheme 风格。程序员通常遵循阻力最小的路径。他们更有可能使用语法成本不大或使代码过于冗长的功能。用它来鼓励你想要的编程风格！我也认为语法灵活非常重要；通过这种方式，库可以有意义地扩展您的语言，让它超越您最初的想法。使用语法的主要部分是理解解析以及如何表示抽象语法树 (AST). 这两个在很大程度上都被视为已解决的问题，尽管确切的解决方案将取决于您用于实现系统的语言。像 Haskell 这样的函数式语言在这类事情上特别擅长，让您将 AST 表示为代数数据类型，并支持非常好的解析库，如 Parsec。语义
语言最重要的部分是它的语义。对于许多语言——尤其是脚本语言——它们的定义通常很差。第一个实现隐含地定义了语言的行为方式。我认为这是一个糟糕的想法：它让设计更复杂、更不优雅的语言变得更加容易，也让推理变得更加困难关于语言的行为。将来，如果您没有某种外部语言规范，您如何知道哪些优化应该被允许，哪些实现实际上是正确的？令人高兴的是，有一些非常有用的工具可用于对语言语义进行建模、设计和推理. 事实上，这是编程语言理论中的一个大领域。第一步是学习操作语义和指称语义，这让您可以正式指定程序的含义. 为你的语言提出语义，即使你不是 100% 正式的，也可能会暴露极端情况——参数的评估顺序是什么？线程系统如何工作？是否有多种正确的方法来评估相同的代码？（最后一个对于优化代码特别重要。）λ-演算
另一个非常有用的语言设计工具是 λ-演算。λ-演算或多或少是最小的编程语言。这使其成为原型功能的绝佳平台. 当您提出一个新想法时，您可以使用 λ-演算模型独立于其他所有内容进行测试，而不是立即尝试将其与您的语言中的其他所有内容相协调。这使您可以完全充实并尝试该想法，而无需使用完整的编程语言所带来的所有复杂性。当然，您还应该考虑任何给定的功能如何适合您的语言的宏伟计划。但是 lambda 演算首先是设计功能的好方法。类型
要考虑的另一件事是您的类型系统。你希望你的语言是静态类型的吗？（我愿意！）如果你这样做，你应该复习一下类型理论. 它本身实际上是一个非常丰富的主题，它为设计语言提供了很好的见解。一本非常好的书是类型和编程语言. 同样，您可以将类型系统的某些部分与语言的其余部分分开设计，方法是将它们分层到像 System F 这样的 λ-演算的类型化版本上，它只有简单的类型和多态性。函数式语言最直接地采用了这种设计：Haskell 基本上是 System F 的修改变体的皮肤。但即使是像 Java 这样的非函数式（不正常的？）语言也有一些特性，比如可变变量和泛型，使用λ-演算。也就是说，确保任何给定的打字规则与您的其他语言相匹配仍然非常重要。ML 实际上有一个这样的问题，其中两个特性（多态性和可变引用）并没有很好地混合。为了避免这种事情，在将功能集成回您的语言本身时要小心。解释器在设计语法、语义和类型时，为您的语言提供一个简单的解释器
确实很有帮助。您可以轻松实现您提出的新功能，允许您将它们用于试驾并查看它们在实践中的样子。您可以从一个简单的 λ-演算解释器开始，并以不同的方式对其进行扩展。但是，我发现（根据经验），在外部解决给定特征的语义非常重要的解释器。不要让解释器真正定义你的语言：它只是一个实现。如果您只是实现一个解释器而没有过多考虑该语言的行为方式，那么很容易将自己编码到一个角落，正如我在不久前自己设计一种语言时发现的那样¹。将您的解释器视为 MVP：您想要最简单的事情就是看到你的语言。不要担心性能或正确性或花哨的工具或任何东西。让它工作。在这一点上，它更像是你自己的工具，而不是外部用户的任何东西。
